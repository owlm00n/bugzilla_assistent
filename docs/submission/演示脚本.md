## 一、课题概述

### 课题名称
**OWLMYCLAW — 面向底层驱动工程师的 AI 调试助手**

### 课题背景
底层驱动工程师（USB/PD/Type-C）日常调试面临三大痛点：
- **日志格式碎片化**：展锐/高通/MTK 各平台日志格式不同，无法用固定正则解析
- **协议知识门槛高**：PD 3.1 协议数百页规范，查表解码耗时
- **Bug 排查链路长**：从 Bugzilla 查 Bug → 下载日志 → 分析定位 → 写调试代码，多工具切换

### 解决痛点
| 痛点 | OWLMYCLAW 方案 |
|------|---------------|
| 多平台日志格式不统一 | AI 语义理解 + 渐进式提取，兼容任意格式 |
| 协议解码依赖人工查表 | 知识库驱动自动解码，力科风格可视化 |
| 日志分析缺乏角色聚焦 | 角色画像感知，自动筛选关注点 |
| 调试关键词散落各处 | 关键词收纳系统，按问题方向分类管理 |
| Bug 排查多工具切换 | Bug 查询→日志下载→分析→代码生成，一站式 |

### 场景 1：protocol-parser — 协议智能解析（2分钟）

#### 1.1 PD 协商失败分析

**对话输入：**
> "帮我分析 owlmyclaw/data/sample_pd_negotiation_fail.log，看看 PD 协商出了什么问题"

1. 调用 `smart_extractor.py` 渐进式提取日志
2. 加载 `pd3.1_spec.md` 知识库
3. AI 语义理解，解码 header hex → 消息类型
4. 解析 PDO 结构，追踪状态机流转
5. 检测异常，生成可视化报告

**展示结果：**
- 力科协议分析仪风格 HTML 报告
- 仪表盘（V/A/W 实时数据）
- 状态机流转图
- 报文时序表格（含 bit 级解码）
- PDO 能力表
- 异常标注（红色高亮）

**口播：**
> "protocol-parser 的核心优势是 AI 驱动——不依赖固定正则。无论展锐、高通还是 MTK 的日志格式，AI 都能理解语义并正确解析。结果以力科协议分析仪风格呈现，包含仪表盘、状态机流转图、报文时序和 bit 级解码。"

### 场景 4：code-generator — 分级代码生成（1分钟）

#### 4.1 基于分析结果生成调试代码

**对话输入：**
> "基于刚才 PD 协商失败的分析结果，生成 Level 2 的诊断代码"

**Agent 执行流程：**
1. 提取异常类型（PD 协商超时）
2. 识别驱动类型（PD/TCPM）
3. 选择代码等级（Level 2 完整诊断）
4. 生成分级代码片段

**展示结果：**
```c
// Level: 2 - 完整诊断
// 驱动类型: PD/TCPM
// 问题描述: PD协商超时，VBUS电压未达到预期值

// === Level 2: 完整诊断 ===
static void tcpm_pd_state_debug(struct tcpm_port *port)
{
    struct pd_message *msg = &port->rx_msg;
    
    // 状态机状态快照
    pr_info("TCPM: state=%s, vbus=%dmV, cc1=%d, cc2=%d",
            tcpm_state_name(port->state),
            tcpm_get_vbus(port),
            tcpm_get_cc(port, 0),
            tcpm_get_cc(port, 1));
    
    // 消息头解码
    if (msg->header) {
        pr_info("TCPM: msg_type=%d, data_role=%d, power_role=%d, msg_id=%d",
                PD_HEADER_TYPE(msg->header),
                PD_HEADER_DR(msg->header),
                PD_HEADER_PR(msg->header),
                PD_HEADER_ID(msg->header));
    }
    
    // 超时检测
    if (ktime_after(ktime_get(), port->timer_expires)) {
        pr_err("TCPM: TIMEOUT! state=%s, expected=%s",
               tcpm_state_name(port->state),
               tcpm_state_name(port->expected_state));
        dump_stack();
    }
}

// === 编译/加载说明 ===
// 1. 将代码添加到 drivers/usb/typec/tcpm/tcpm.c
// 2. 重新编译内核模块: make M=drivers/usb/typec
// 3. 加载模块: insmod tcpm.ko
// 4. 查看日志: dmesg -w | grep TCPM

// === 回归测试建议 ===
// 1. 连接 PD 充电器，观察协商流程
// 2. 验证 VBUS 电压在预期范围内
// 3. 测试 PR_SWAP 和 DR_SWAP 场景
```

**口播：**
> "code-generator 提供三级分级代码——Level 1 快速调试只需 5 分钟，Level 2 完整诊断 15 分钟，Level 3 生产级修复。基于分析结果自动识别驱动类型，生成符合内核编码规范的代码。"
